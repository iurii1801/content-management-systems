# Лабораторная работа №4. Разработка плагина для WordPress

## Цель работы

Освоить расширяемую модель данных WordPress: создать CPT (Custom Post Type), пользовательскую таксономию, метаданные с метабоксом в админ-панели, а также реализовать виджет для отображения данных на сайте.

## Условие

Собрать учебный плагин USM Notes, который добавляет в сайт раздел «Заметки» с приоритетами и датой напоминания.

## Ход работы

### Шаг 3. Регистрация Custom Post Type (CPT)

На данном этапе необходимо реализовать пользовательский тип записей (**Custom Post Type**) для хранения заметок в рамках разрабатываемого плагина **USM Notes**.

Использование пользовательских типов записей позволяет расширить стандартную модель данных `WordPress` и создать отдельную сущность, предназначенную для хранения специфических данных приложения.

В рамках лабораторной работы создаётся тип записей **Notes**, который будет использоваться для создания и управления заметками.

#### Создание функции регистрации CPT

Для регистрации пользовательского типа записей необходимо использовать встроенную функцию `register_post_type()`.

В основном файле плагина `usm-notes.php` создаётся функция, которая выполняет регистрацию нового типа записей.

В данной функции определяется набор меток (`labels`), используемых системой `WordPress` для отображения элементов интерфейса в административной панели.

Также задаются основные параметры пользовательского типа записей.

Код регистрации CPT:

```php
/**
 * Регистрация Custom Post Type "Notes"
 */
function usm_register_notes_cpt() {

    $labels = array(
        'name'               => 'Notes',
        'singular_name'      => 'Note',
        'menu_name'          => 'Notes',
        'name_admin_bar'     => 'Note',
        'add_new'            => 'Add New',
        'add_new_item'       => 'Add New Note',
        'new_item'           => 'New Note',
        'edit_item'          => 'Edit Note',
        'view_item'          => 'View Note',
        'all_items'          => 'All Notes',
        'search_items'       => 'Search Notes',
        'not_found'          => 'No notes found',
        'not_found_in_trash' => 'No notes found in Trash'
    );

    $args = array(
        'labels'       => $labels,
        'public'       => true,
        'has_archive'  => true,
        'menu_icon'    => 'dashicons-edit-page',
        'supports'     => array('title', 'editor', 'author', 'thumbnail'),
        'show_in_rest' => true
    );

    register_post_type('usm_note', $args);
}
```

![image](https://i.imgur.com/ZDhqaRF.png)

#### Настройка параметров пользовательского типа записей

В параметрах регистрации задаются основные характеристики нового типа записей:

* `public => true` — делает тип записей публичным и доступным как в административной панели, так и на сайте.
* `has_archive => true` — включает автоматическое создание архивной страницы для всех заметок.
* `menu_icon` — задаёт иконку раздела в административном меню WordPress.
* `supports` — определяет набор поддерживаемых элементов редактора (заголовок, текст записи, автор и миниатюра).
* `show_in_rest => true` — обеспечивает поддержку REST API и современного редактора Gutenberg.

#### Подключение регистрации CPT через хук init

Для того чтобы регистрация типа записей происходила при запуске системы `WordPress`, созданная функция подключается к хуку `init`.

Для этого в файл плагина добавляется следующая строка:

```php
add_action('init', 'usm_register_notes_cpt');
```

Данный хук выполняется на этапе инициализации системы, что позволяет корректно зарегистрировать новый тип записей до начала обработки страницы.

После сохранения файла и обновления административной панели `WordPress` в боковом меню появляется новый раздел **Notes**, предназначенный для управления заметками.

### Шаг 4. Регистрация пользовательской таксономии

На данном этапе необходимо реализовать пользовательскую таксономию (**taxonomy**), которая будет использоваться для классификации заметок по уровню приоритета.

Таксономии в `WordPress` позволяют группировать записи по определённым категориям или характеристикам. В рамках лабораторной работы создаётся таксономия **Priority**, предназначенная для указания приоритета заметки.

Данная таксономия будет связана с ранее созданным пользовательским типом записей **Notes**.

#### Создание функции регистрации таксономии

Для регистрации пользовательской таксономии используется встроенная функция `register_taxonomy()`.

В основном файле плагина `usm-notes.php` создаётся функция регистрации таксономии **Priority**, в которой определяются метки (`labels`) и параметры таксономии.

Код реализации:

```php
/**
 * Регистрация таксономии "Priority"
 */
function usm_register_priority_taxonomy() {

    $labels = array(
        'name'          => 'Priorities',
        'singular_name' => 'Priority',
        'search_items'  => 'Search Priorities',
        'all_items'     => 'All Priorities',
        'edit_item'     => 'Edit Priority',
        'update_item'   => 'Update Priority',
        'add_new_item'  => 'Add New Priority',
        'new_item_name' => 'New Priority Name',
        'menu_name'     => 'Priority'
    );

    $args = array(
        'hierarchical'  => true,
        'labels'        => $labels,
        'public'        => true,
        'show_in_rest'  => true
    );

    register_taxonomy('usm_priority', array('usm_note'), $args);
}

add_action('init', 'usm_register_priority_taxonomy');
```

![image](https://i.imgur.com/Lwl923i.png)

#### Настройка параметров таксономии

В параметрах регистрации указываются основные характеристики таксономии:

* `hierarchical => true` — делает таксономию иерархической, аналогично категориям WordPress;
* `public => true` — таксономия является публичной и доступной в административной панели;
* `labels` — набор текстовых меток, используемых системой WordPress для отображения элементов интерфейса;
* `show_in_rest => true` — обеспечивает поддержку REST API и редактора Gutenberg.

Также в функции `register_taxonomy()` указывается связь таксономии с пользовательским типом записей:

```
usm_note
```

Это означает, что таксономия **Priority** будет использоваться для классификации заметок.

#### Регистрация таксономии через хук init

Для корректной работы таксономии необходимо зарегистрировать её при инициализации системы WordPress.

Для этого используется хук `init`:

```php
add_action('init', 'usm_register_priority_taxonomy');
```

При запуске WordPress функция регистрации таксономии выполняется автоматически, что позволяет системе корректно добавить новую структуру данных.

### Шаг 5. Добавление метабокса для даты напоминания

На данном этапе необходимо реализовать дополнительное метаполе для пользовательского типа записей **Notes**, которое будет хранить дату напоминания.

Для этого в редактор записей добавляется специальный **метабокс**, содержащий поле выбора даты. Также реализуется механизм сохранения значения, проверка корректности введённых данных и отображение даты в административной панели.

#### Добавление метабокса в редактор заметок

Для добавления дополнительного блока в редактор записей используется функция `add_meta_box()`.

Создаётся функция, которая регистрирует метабокс **Due Date** для пользовательского типа записей `usm_note`.

Код добавления метабокса:

```php
/**
 * Добавление метабокса для даты напоминания
 */
function usm_add_due_date_meta_box() {
    add_meta_box(
        'usm_due_date_meta_box',
        'Due Date',
        'usm_render_due_date_meta_box',
        'usm_note',
        'side',
        'default'
    );
}
add_action('add_meta_boxes', 'usm_add_due_date_meta_box');
```

В данном фрагменте:

* `add_meta_box()` регистрирует новый метабокс;
* `'usm_due_date_meta_box'` — уникальный идентификатор блока;
* `'Due Date'` — заголовок блока;
* `'usm_render_due_date_meta_box'` — функция, отвечающая за вывод содержимого;
* `'usm_note'` — тип записи, к которому относится метабокс;
* `'side'` — расположение метабокса в правой колонке редактора.

Метабокс автоматически появляется на странице создания и редактирования заметок.

#### Отрисовка поля выбора даты

Внутри метабокса необходимо вывести поле выбора даты. Для этого используется HTML5-элемент `input type="date"`.

Код отрисовки метабокса:

```php
/**
 * Отрисовка метабокса
 */
function usm_render_due_date_meta_box($post) {
    wp_nonce_field('usm_save_due_date', 'usm_due_date_nonce');

    $due_date = get_post_meta($post->ID, '_usm_due_date', true);

    echo '<label for="usm_due_date_field">Select reminder date:</label>';
    echo '<input type="date" id="usm_due_date_field" name="usm_due_date_field" value="' . esc_attr($due_date) . '" style="width:100%;" required />';
}
```

В данном коде:

* `wp_nonce_field()` создаёт специальный **nonce-токен**, который защищает сохранение данных от подделки запросов;
* `get_post_meta()` получает ранее сохранённое значение даты;
* `input type="date"` выводит поле выбора даты с календарём;
* `esc_attr()` безопасно выводит значение даты в HTML.

При повторном редактировании записи ранее сохранённая дата автоматически отображается в поле.

#### Сохранение значения даты

Для сохранения значения метаполя используется хук `save_post`, который срабатывает при сохранении записи.

Код сохранения:

```php
/**
 * Сохранение даты напоминания
 */
function usm_save_due_date_meta($post_id) {

    if (!isset($_POST['post_type']) || $_POST['post_type'] !== 'usm_note') {
        return;
    }

    if (!isset($_POST['usm_due_date_nonce'])) {
        return;
    }

    if (!wp_verify_nonce($_POST['usm_due_date_nonce'], 'usm_save_due_date')) {
        return;
    }

    if (defined('DOING_AUTOSAVE') && DOING_AUTOSAVE) {
        return;
    }

    if (!current_user_can('edit_post', $post_id)) {
        return;
    }

    $due_date = isset($_POST['usm_due_date_field']) ? sanitize_text_field($_POST['usm_due_date_field']) : '';
```

В данном фрагменте выполняются проверки:

* проверяется тип записи (`usm_note`);
* проверяется наличие `nonce`;
* `wp_verify_nonce()` защищает от подделки формы;
* `DOING_AUTOSAVE` предотвращает сохранение при автосохранении;
* `current_user_can()` проверяет права пользователя;
* `sanitize_text_field()` очищает введённое значение.

#### Проверка обязательности поля и валидация даты

После получения значения выполняется проверка, что поле заполнено и дата не находится в прошлом.

```php
    if (empty($due_date)) {
        delete_post_meta($post_id, '_usm_due_date');
        set_transient('usm_due_date_error_' . get_current_user_id(), 'empty', 30);
        return;
    }

    $today = date('Y-m-d');

    if ($due_date < $today) {
        delete_post_meta($post_id, '_usm_due_date');
        set_transient('usm_due_date_error_' . get_current_user_id(), 'past', 30);
        return;
    }

    update_post_meta($post_id, '_usm_due_date', $due_date);
}

add_action('save_post', 'usm_save_due_date_meta');
```

Здесь:

* `empty($due_date)` проверяет обязательность заполнения поля;
* `date('Y-m-d')` получает текущую дату;
* сравнение `$due_date < $today` проверяет, что дата не находится в прошлом;
* `update_post_meta()` сохраняет дату в метаполе записи.

#### Вывод сообщений об ошибках

Если дата не указана или указана дата в прошлом, пользователю выводится сообщение об ошибке.

```php
/**
 * Вывод сообщений об ошибках
 */
function usm_due_date_admin_notices() {

    $error = get_transient('usm_due_date_error_' . get_current_user_id());

    if (!$error) {
        return;
    }

    if ($error === 'empty') {
        echo '<div class="notice notice-error is-dismissible"><p>Due Date is required.</p></div>';
    }

    if ($error === 'past') {
        echo '<div class="notice notice-error is-dismissible"><p>Due Date cannot be in the past.</p></div>';
    }

    delete_transient('usm_due_date_error_' . get_current_user_id());
}

add_action('admin_notices', 'usm_due_date_admin_notices');
```

Здесь:

* `set_transient()` временно сохраняет информацию об ошибке;
* `admin_notices` выводит сообщение в административной панели;
* `delete_transient()` удаляет сообщение после отображения.

Если дата сохранена, она отображается в колонке **Due Date**, иначе выводится символ `—`.

### Шаг 6. Создание шорткода для отображения заметок

На данном этапе необходимо реализовать шорткод, позволяющий выводить заметки пользовательского типа **Notes** на страницах сайта.

Шорткод должен поддерживать фильтрацию по приоритету и по дате напоминания, а также выводить все заметки, если параметры фильтрации не заданы.

В рамках лабораторной работы реализуется шорткод следующего вида:

```php
[usm_notes priority="X" before_date="YYYY-MM-DD"]
```

где:

* `priority` — фильтр по значению таксономии **Priority**;
* `before_date` — фильтр по дате напоминания **Due Date**.

#### Создание функции обработки шорткода

Для обработки шорткода создаётся отдельная функция `usm_notes_shortcode()`, которая принимает атрибуты шорткода, формирует запрос к базе данных и выводит подходящие записи.

Код реализации:

```php
/**
 * Шорткод для вывода заметок
 */
function usm_notes_shortcode($atts) {
    $atts = shortcode_atts(array(
        'priority'    => '',
        'before_date' => ''
    ), $atts, 'usm_notes');

    $args = array(
        'post_type'      => 'usm_note',
        'post_status'    => 'publish',
        'posts_per_page' => -1
    );

    $meta_query = array();
    $tax_query = array();

    if (!empty($atts['before_date'])) {
        $meta_query[] = array(
            'key'     => '_usm_due_date',
            'value'   => sanitize_text_field($atts['before_date']),
            'compare' => '<=',
            'type'    => 'DATE'
        );
    }

    if (!empty($atts['priority'])) {
        $tax_query[] = array(
            'taxonomy' => 'usm_priority',
            'field'    => 'slug',
            'terms'    => sanitize_title($atts['priority'])
        );
    }

    if (!empty($meta_query)) {
        $args['meta_query'] = $meta_query;
    }

    if (!empty($tax_query)) {
        $args['tax_query'] = $tax_query;
    }

    $query = new WP_Query($args);

    ob_start();

    echo '<div class="usm-notes-list">';

    if ($query->have_posts()) {
        while ($query->have_posts()) {
            $query->the_post();

            $due_date = get_post_meta(get_the_ID(), '_usm_due_date', true);
            $terms = get_the_terms(get_the_ID(), 'usm_priority');

            echo '<div class="usm-note-item">';
            echo '<h3>' . get_the_title() . '</h3>';
            echo '<div class="usm-note-content">' . get_the_excerpt() . '</div>';

            if ($terms && !is_wp_error($terms)) {
                echo '<p><strong>Priority:</strong> ' . esc_html($terms[0]->name) . '</p>';
            }

            if ($due_date) {
                echo '<p><strong>Due Date:</strong> ' . esc_html($due_date) . '</p>';
            }

            echo '</div>';
        }
    } else {
        echo '<p>Нет заметок с заданными параметрами</p>';
    }

    echo '</div>';

    wp_reset_postdata();

    return ob_get_clean();
}
add_shortcode('usm_notes', 'usm_notes_shortcode');
```

В данном коде:

* `shortcode_atts()` задаёт значения атрибутов по умолчанию;
* `WP_Query` используется для получения заметок типа `usm_note`;
* `tax_query` применяется для фильтрации по приоритету;
* `meta_query` применяется для фильтрации по дате напоминания;
* `get_post_meta()` получает значение даты напоминания;
* `get_the_terms()` получает назначенный приоритет;
* если записи не найдены, выводится сообщение:
  **«Нет заметок с заданными параметрами»**.

#### Логика фильтрации данных

В функции предусмотрены два вида фильтрации:

1. **Фильтрация по приоритету**
   Если в шорткоде указан параметр `priority`, в запрос добавляется `tax_query`, ограничивающий выборку по таксономии `usm_priority`.

2. **Фильтрация по дате**
   Если указан параметр `before_date`, в запрос добавляется `meta_query`, выбирающий записи, у которых значение `_usm_due_date` меньше или равно заданной дате.

Если оба параметра отсутствуют, в запрос не добавляются дополнительные условия, и шорткод выводит **все заметки**.

#### Регистрация шорткода

После создания функции выполняется регистрация шорткода с помощью:

```php

```

Это позволяет использовать шорткод `[usm_notes]` на любой странице или записи WordPress.

## Контрольные вопросы

### 1. Чем пользовательская таксономия принципиально отличается от метаполя? Приведи пример, когда выбрать таксономию, а когда — метаданные

В WordPress пользовательская **таксономия** и **метаполе** используются для хранения дополнительной информации о записях, но выполняют разные задачи.

**Таксономия** используется для **классификации и группировки записей**. Она позволяет объединять записи по определённым категориям или тегам и затем фильтровать или выводить их по этим значениям.

Например, в плагине заметок используется таксономия **Priority**, которая имеет значения:

* High
* Medium
* Low

Это позволяет легко группировать заметки по уровню важности и фильтровать их на сайте.

**Метаполе** используется для хранения **дополнительных данных конкретной записи**, которые не предназначены для группировки.

Например, поле **Due Date** хранит дату напоминания для каждой заметки. Это значение индивидуально для каждой записи и используется для отображения или фильтрации по дате.

Таким образом:

* **таксономии** применяются для классификации записей
* **метаполя** — для хранения дополнительных характеристик записи

### 2. Зачем нужен nonce при сохранении метаполей и что произойдёт, если его не проверять?

**Nonce** (number used once) используется в WordPress для защиты формы от несанкционированных запросов.

При сохранении метаполей nonce добавляется в форму и затем проверяется при обработке данных. Это позволяет убедиться, что запрос на сохранение данных был отправлен именно из административной панели WordPress.

Проверка nonce обычно выполняется с помощью функции:

```
wp_verify_nonce()
```

Если nonce не проверять, возникает риск **CSRF-атаки (Cross-Site Request Forgery)**. В этом случае злоумышленник может отправить запрос на изменение данных от имени пользователя, который авторизован в системе.

В результате без проверки nonce могут быть изменены или добавлены данные записей без ведома администратора.

Таким образом, использование nonce обеспечивает **безопасность при сохранении метаполей**.

### 3. Какие аргументы `register_post_type()` и `register_taxonomy()` чаще всего важны для фронтенда и UX (назови минимум три и объясни почему)

При регистрации пользовательских типов записей и таксономий в WordPress используются различные аргументы, которые влияют на отображение и удобство работы с ними.

Одним из важных аргументов является **`public`**.
Он определяет, будет ли тип записи доступен на фронтенде сайта. Если значение установлено в `true`, записи можно просматривать на сайте.

Второй важный аргумент — **`has_archive`**.
Он отвечает за создание архивной страницы для типа записи. Это позволяет пользователям просматривать список всех записей данного типа на отдельной странице.

Третий аргумент — **`rewrite`**.
Этот параметр используется для настройки структуры URL. Благодаря ему можно создавать понятные и удобные адреса страниц, например:

```
site.com/notes/название-записи
```

Также важным является аргумент **`supports`**, который определяет, какие элементы будут доступны в редакторе записи. Например:

* заголовок
* редактор текста
* миниатюра

Это влияет на удобство работы пользователей с системой управления контентом.

Таким образом, правильная настройка аргументов при регистрации типов записей и таксономий позволяет улучшить **структуру сайта, удобство работы администратора и отображение данных на фронтенде**.
