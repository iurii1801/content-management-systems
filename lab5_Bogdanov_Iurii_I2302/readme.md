# Лабораторная работа №5. Безопасность WordPress

## Цель работы

Закрепить ключевые практики безопасности WordPress: управление ролями и паролями, обновления, базовое hardening (wp-config.php, права, отключение редактора), резервное копирование, мониторинг активности и поэтапная настройка All In One WP Security & Firewall (AIOS) для защиты от брутфорса, базового WAF и контроля прав.

## Ход работы

### Шаг 1. Подготовка среды

Для выполнения лабораторной работы используется ранее установленная система `WordPress`.

Необходимо перейти в административную панель сайта через браузер по адресу:

```
http://localhost:8090/wp_lab2/wp-admin
```

После перехода выполняется вход под пользователем с правами администратора, что обеспечивает доступ ко всем функциям системы.

![image](https://i.imgur.com/PqVAO98.png)

Также для настройки и контроля безопасности необходимо включить режим отладки `WordPress`.

Для этого открывается конфигурационный файл:

```
wp-config.php
```

В данном файле необходимо установить следующий параметр:

```php
define('WP_DEBUG', true);
```

Включение режима отладки позволяет отслеживать ошибки и предупреждения системы, что важно при настройке безопасности и тестировании.

### Шаг 2. Управление ролями и паролями

#### Создание тестового пользователя с ролью `Автор`

Для проверки разграничения прав доступа необходимо создать тестового пользователя с ролью `Автор`.

Для этого в административной панели `WordPress` выполняется переход в раздел:

```
Пользователи → Добавить нового пользователя
```

![image](https://i.imgur.com/S8GYjPX.png)

При создании пользователя указываются логин, адрес электронной почты, имя, фамилия и задаётся надёжный пароль. В качестве роли выбирается `Автор`.

Роль `Автор` позволяет пользователю создавать, редактировать и публиковать только собственные записи, не предоставляя доступа к административным настройкам сайта.

![image](https://i.imgur.com/08l9bNj.png)

После создания учётной записи необходимо убедиться, что пользователь успешно добавлен в систему. Для этого выполняется переход в раздел:

```
Пользователи → Все пользователи
```

В списке пользователей отображаются все зарегистрированные учётные записи, включая администратора и созданного тестового пользователя.

![image](https://i.imgur.com/tdLmJw0.png)

Далее необходимо проверить надёжность пароля администратора. Для этого выполняется переход в раздел:

```
Пользователи → Профиль
```

Пароль должен соответствовать требованиям безопасности: содержать не менее 8 символов, включая буквы, цифры и специальные символы. Использование сложных паролей снижает риск несанкционированного доступа к системе.

### Шаг 4. Базовое hardening

Для повышения уровня безопасности `WordPress` необходимо выполнить базовые меры hardening, включающие запрет редактирования файлов через административную панель, настройку прав доступа к файлам и каталогам, а также защиту конфигурационного файла `wp-config.php`.

#### Запрет редактирования файлов из админки

Для отключения встроенного редактора файлов `WordPress` необходимо открыть файл:

```php
wp-config.php
```

В данном файле необходимо добавить строку:

```php
define('DISALLOW_FILE_EDIT', true);
```

Данный параметр запрещает редактирование файлов тем и плагинов через административную панель, что снижает риск внедрения вредоносного кода в случае получения злоумышленником доступа к учётной записи администратора.

![image](https://i.imgur.com/NXW1ocE.png)

#### Настройка прав доступа к файлам и папкам

Для обеспечения безопасной работы системы необходимо установить рекомендуемые права доступа:

* для каталогов — `755`
* для файлов — `644`

Настройка прав доступа выполняется из терминала с помощью следующих команд:

```bash
chmod -R 755 .
find . -type f -exec chmod 644 {} \;
```

![image](https://i.imgur.com/ochdsJx.png)

Первая команда:

```bash
chmod -R 755 .
```

рекурсивно устанавливает права доступа `755` для всех файлов и каталогов в текущей директории. Это означает, что владелец имеет полный доступ (чтение, запись, выполнение), а остальные пользователи — только чтение и выполнение.

Вторая команда:

```bash
find . -type f -exec chmod 644 {} \;
```

находит все файлы в текущей директории и устанавливает для них права `644`. Это означает, что владелец имеет права на чтение и запись, а остальные пользователи — только на чтение. Данная команда используется для корректировки прав после первой команды, чтобы файлы не имели избыточных прав на выполнение.

После выполнения команд необходимо выполнить проверку установленных прав доступа с помощью следующих команд:

```bash
ls -l wp-config.php
ls -ld wp-content
```

Команда:

```bash
ls -l wp-config.php
```

позволяет проверить права доступа для файла `wp-config.php`. Результат вида:

```
-rw-r--r--
```

соответствует правам `644`.

Команда:

```bash
ls -ld wp-content
```

используется для проверки прав доступа каталога `wp-content`. Результат вида:

```
drwxr-xr-x
```

соответствует правам `755`.

![image](https://i.imgur.com/LZR76B0.png)

Таким образом подтверждается корректность установки прав доступа к файлам и каталогам.

#### Защита файла wp-config.php через .htaccess

Для ограничения прямого доступа к конфигурационному файлу `WordPress` необходимо открыть файл:

```apache
.htaccess
```

В конец данного файла необходимо добавить следующий блок:

```apache
<files wp-config.php>
    order allow,deny
    deny from all
</files>
```

![image](https://i.imgur.com/iN37elt.png)

Добавленный блок предназначен для запрета доступа к файлу `wp-config.php` через веб-сервер, что позволяет дополнительно защитить конфиденциальные параметры сайта.

### Шаг 5. Установка и первичная настройка All In One WP Security & Firewall (AIOS)

#### Установка и активация плагина

В административной панели WordPress осуществляется переход в раздел **«Плагины» → «Добавить новый»**.
В строке поиска вводится название плагина **All In One WP Security & Firewall**.

После нахождения плагина выполняется его установка и последующая активация.

![image](https://i.imgur.com/ffVbW5F.png)

#### Переход в раздел настроек плагина

После активации плагина в боковом меню административной панели появляется пункт **AIOS**.
Осуществляется переход в раздел **«AIOS» → «Консоль»**, откуда доступны все настройки безопасности.

![image](https://i.imgur.com/cNDIxsy.png)

#### User Login

В административной панели осуществляется переход в раздел
**«AIOS» → «Безопасность пользователя» → «Блокировка входа в систему»**.

Активируется функция ограничения попыток входа (**Login Lockdown**).
Устанавливаются следующие параметры:

* максимальное количество попыток входа: **5**
* период времени для попыток: **15 минут**
* время блокировки: **30 минут**

Дополнительно включается функция принудительного выхода пользователей из системы.
Для этого выполняется переход в раздел
**«AIOS» → «Безопасность пользователя» → «Принудительный выход из системы»**,
где задаётся время автоматического выхода — **24 часа (1440 минут)**.

![image](https://i.imgur.com/hrF17T7.png)

![image](https://i.imgur.com/qR1jput.png)

Проверяется наличие пользователя с именем **admin**.

#### Filesystem Security

В административной панели осуществляется переход в раздел
**«AIOS» → «Безопасность файлов» → «Права доступа к файлам»**.

В данном разделе выполняется проверка прав доступа к файлам и каталогам WordPress.
Плагин анализирует текущие разрешения файловой системы и выявляет потенциальные уязвимости.

В рассматриваемом случае система определяет, что сайт работает на сервере под управлением **Windows**,
в связи с чем некоторые функции проверки и изменения прав доступа (характерные для Linux-систем) недоступны.

Таким образом, изменение прав доступа не выполняется, однако проверка состояния безопасности файлов считается выполненной.

![image](https://i.imgur.com/sy6ofaB.png)

#### Firewall

В административной панели осуществляется переход в раздел
**«AIOS» → «Брандмауэр» → «Правила PHP»**.

Активируется базовый уровень защиты брандмауэра (**Basic Firewall**), обеспечивающий фильтрацию входящих запросов и защиту от распространённых уязвимостей.

![image](https://i.imgur.com/RlUbl3k.png)

Далее осуществляется переход в раздел
**«AIOS» → «Брандмауэр» → «Безопасность URL-адресов»**.

Включается параметр **отключения неверных строк запроса (Bad Query Strings)**, который блокирует подозрительные URL-запросы, содержащие вредоносные или некорректные данные.

Данный механизм позволяет предотвратить попытки XSS-атак и внедрения вредоносного кода через адресную строку.

![image](https://i.imgur.com/SSHur96.png)

Далее осуществляется переход в раздел
**«AIOS» → «Брандмауэр» → «Фильтрация строк»**.

Включается расширенный фильтр символьных строк, обеспечивающий защиту от XSS-атак (**Cross-Site Scripting**).

![image](https://i.imgur.com/p1czqqB.png)

Также осуществляется переход в раздел
**«AIOS» → «Брандмауэр» → «Правила .htaccess» → «Содержимое списка каталога»**.

Включается параметр отключения отображения содержимого каталогов (**directory browsing**), что предотвращает просмотр структуры файлов сайта.

![image](https://i.imgur.com/Rva99cI.png)

Таким образом, на сайте активирован базовый уровень брандмауэра и включены основные механизмы защиты от распространённых атак.

#### Brute Force

В административной панели осуществляется переход в раздел
**«AIOS» → «Брутфорс» → «Переименование страницы входа»**.

Активируется функция изменения URL страницы входа (**Rename Login Page**), что позволяет скрыть стандартный адрес `/wp-login.php` и тем самым снизить риск автоматизированных атак перебора паролей (brute force).

В поле URL задаётся новое значение страницы входа, например:
**`iurii-login`**, в результате чего адрес входа изменяется на нестандартный.

После внесения изменений настройки сохраняются.

![image](https://i.imgur.com/twnF4Fz.png)

После активации функции проверяется доступ к странице входа по новому адресу.
Страница авторизации успешно открывается по изменённому URL, что подтверждает корректность настройки.

![image](https://i.imgur.com/PXrxapH.png)

Изменение стандартного URL страницы входа повышает безопасность сайта, так как затрудняет автоматический подбор паролей и скрывает точку входа от потенциальных злоумышленников.

#### Scanner / Malware

В административной панели осуществляется переход в раздел
**«AIOS» → «Сканер» → «Обнаружение изменения файлов»**.

Активируется функция автоматического сканирования файлов (**File Change Detection**), которая позволяет отслеживать изменения в файлах сайта.

Задаётся интервал сканирования (например, **4 недели**), после чего система будет периодически проверять файлы на наличие изменений.

Дополнительно включается параметр отправки уведомлений на электронную почту при обнаружении изменений.
В поле указывается адрес электронной почты администратора сайта.

![image](https://i.imgur.com/sCgqMma.png)

Функция отслеживания изменений файлов позволяет своевременно выявлять несанкционированные изменения, что повышает уровень безопасности сайта и помогает оперативно реагировать на возможные угрозы.

### Шаг 6. Проверка защиты от брутфорса

Для проверки работы механизма защиты от перебора паролей (Brute Force) выполнена серия неудачных попыток входа в систему.

Сначала выполнен выход из административной панели WordPress. Далее осуществлён переход на новый URL страницы входа в систему:

```
http://localhost:8090/wp_lab2/iurii-login
```

После этого введено неверное значение пароля более 5 раз подряд.

В результате сработал механизм защиты **Login Lockdown**, и доступ с текущего IP-адреса автоматически заблокируется. При попытке входа появилось сообщение о блокировке:

![image](https://i.imgur.com/pAUK9cm.png)

Данное сообщение подтверждает, что система успешно обнаружила попытку перебора пароля и ограничила доступ.

Далее для анализа блокировки открыт раздел административной панели:

**AIOS → Консоль → Блокировка IP-адресов**

В данном разделе обнаружена запись о блокировке, содержащая следующую информацию:

* IP-адрес пользователя
* имя пользователя (**iurii_admin**)

## Контрольные вопросы

### 1. Почему `DISALLOW_FILE_EDIT` и правильные права на `wp-config.php` существенно уменьшают риск пост-эксплойта?

Параметр **`DISALLOW_FILE_EDIT`** и корректные права доступа к файлу **`wp-config.php`** являются важными мерами защиты WordPress после возможного взлома (пост-эксплойта).

Константа **`DISALLOW_FILE_EDIT`** отключает встроенный редактор файлов в административной панели WordPress. Это предотвращает возможность изменения файлов темы или плагинов через админку.

Если злоумышленник получит доступ к административной панели, он не сможет внедрить вредоносный код напрямую через интерфейс WordPress.

Файл **`wp-config.php`** содержит критически важную информацию:

* данные подключения к базе данных
* секретные ключи безопасности
* настройки системы

При установке правильных прав доступа (например, только для чтения или ограниченный доступ) снижается риск изменения этого файла злоумышленником.

Таким образом:

* `DISALLOW_FILE_EDIT` предотвращает изменение файлов через админку
* защита `wp-config.php` ограничивает доступ к критическим настройкам

Это существенно уменьшает последствия возможного взлома и усложняет закрепление злоумышленника в системе.

### 2. Какие параметры Login Lockdown/Firewall вы выбрали и почему именно такие (обоснуйте баланс безопасности и UX)?

В рамках настройки безопасности были выбраны параметры механизма **Login Lockdown** и базового **Firewall**, обеспечивающие баланс между защитой и удобством использования.

Для Login Lockdown установлены следующие значения:

* максимальное количество попыток входа: **5**
* период попыток: **15 минут**
* время блокировки: **30 минут**

Такие параметры позволяют эффективно защититься от перебора паролей (brute force), при этом не создавая серьёзных неудобств для пользователя при случайной ошибке ввода.

Например, 5 попыток достаточно для обычного пользователя, чтобы исправить ошибку, но недостаточно для эффективной атаки.

В настройках Firewall включены:

* защита от **Bad Query Strings**
* фильтрация строк (защита от **XSS**)
* отключение **directory browsing**

Эти механизмы блокируют распространённые типы атак, не влияя на обычную работу сайта.

Таким образом:

* Login Lockdown ограничивает попытки входа
* Firewall защищает от уязвимостей на уровне запросов

Выбранные параметры обеспечивают **надёжную защиту без ухудшения пользовательского опыта (UX)**.

### 3. Чем отличаются меры защиты на уровне WordPress (плагин/WAF) от мер на уровне веб-сервера и ОС?

Меры защиты WordPress и меры на уровне сервера/операционной системы работают на разных уровнях и дополняют друг друга.

Защита на уровне WordPress (плагин или WAF):

* работает внутри CMS
* контролирует вход пользователей, запросы и действия
* защищает от атак типа brute force, XSS, SQL-инъекций

Например, плагин безопасности может:

* блокировать IP при попытках входа
* фильтровать вредоносные URL-запросы
* отслеживать изменения файлов

Защита на уровне веб-сервера и ОС:

* работает ниже уровня WordPress
* управляет доступом к файлам и сети
* защищает всю систему, а не только сайт

Примеры:

* настройки `.htaccess` или Nginx
* firewall на уровне сервера
* права доступа к файлам
* обновления ОС

Таким образом:

* WordPress-защита — это уровень приложения
* сервер и ОС — это уровень инфраструктуры

Совместное использование этих мер обеспечивает **многоуровневую (layered) безопасность системы**.

### 4. Что обязательно включать в "полный" бэкап WP и как вы проверяете, что восстановление действительно работает?

Полный резервный бэкап WordPress должен включать все компоненты, необходимые для полного восстановления сайта.

Обязательно необходимо включать:

* **базу данных (Database)** — содержит записи, пользователей и настройки
* **загрузки (Uploads)** — изображения и медиафайлы
* **плагины (Plugins)** — функциональность сайта
* **темы (Themes)** — внешний вид сайта

Без любого из этих компонентов восстановление будет неполным.

Проверка корректности восстановления выполняется практическим способом.

Для этого:

1. удаляются данные (например, запись и изображение)
2. выполняется восстановление из резервной копии
3. проверяется результат

После восстановления необходимо убедиться, что:

* удалённая запись снова отображается
* изображение возвращено в медиабиблиотеку

Если данные полностью восстановлены, это подтверждает, что резервная копия является корректной и пригодной для использования.

Таким образом, проверка выполняется через **реальный сценарий восстановления**, что является наиболее надёжным способом убедиться в работоспособности бэкапа.
